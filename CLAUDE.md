# 项目语言规范

## 简体中文要求

在本项目的所有工作中，**必须使用简体中文**，具体规则如下：

### 使用中文
- **聊天回复**：与用户的所有对话使用简体中文
- **思考过程**：`<thinking>` 标签内的推理过程使用简体中文
- **代码注释**：所有注释使用简体中文（包括 `//`、`/* */`、`/** */`、`#` 等）
- **Git 提交信息**：使用简体中文
- **文档文件**：Markdown、README 等使用简体中文
- **标记注释**：TODO、FIXME、NOTE 等使用中文描述

### 例外（保留英文）
- 代码标识符（变量名、函数名、类名等）
- 通用技术术语（API、HTTP、JSON、SQL 等）
- 面向开发者的日志输出
- 配置键值、环境变量名
- 文件路径、URL
- 第三方库/框架名称
- 正则表达式
- 命令行指令及参数

### 优先级
此中文规范为最高优先级，与其他格式规范冲突时以此为准。

# 代码注释规范

## 详细注释要求

在本项目的所有代码中，**必须添加详细的中文注释**，具体规则如下：

### 必须注释的内容

- **文件头部**：每个源文件必须包含文件级注释，说明该文件的用途、职责边界、在架构中的位置
- **类/接口/类型定义**：说明其用途、使用场景、关键设计决策
- **公开方法/函数**：说明功能、参数含义、返回值、使用示例（复杂方法）、前置条件与副作用
- **复杂算法/业务逻辑**：逐步解释算法思路，标注关键分支和边界条件
- **非显而易见的代码**：「为什么这么做」而非「做了什么」——解释设计意图和背景原因
- **魔法数字/硬编码值**：解释数值的来源和含义
- **正则表达式**：说明匹配模式及其目的
- **状态机/生命周期**：说明状态转换条件和各状态含义
- **性能优化点**：解释采用了什么优化策略以及为什么有效
- **HACK/临时方案**：用 `// HACK:` 标记并说明不得已而为之的原因

### 注释密度标准

- 类/接口定义：必须有 JSDoc 风格注释
- 每个方法/函数必须有注释，复杂函数每 10-20 行应有行内注释
- 分支条件（if/switch）中非显而易见的条件应有注释
- 关键变量声明应有注释说明其用途

### 注释风格

- `/** JSDoc */` — 用于类、接口、类型、公开方法、模块导出
- `// 行注释` — 用于行内解释、局部变量、分支条件
- `/* 块注释 */` — 用于需要多行说明的复杂逻辑

### 例外（可简化注释）

- 极其简单的 getter/setter（仅返回/设置单一属性）
- 标准的 React 组件 render 方法
- 已经通过命名自解释的简单单行代码
- 纯样式定义（CSS-in-JS 对象）

### 优先级
此注释规范与中文规范同为最高优先级。
